# Развёртывание серверной части CarCare360 (Docker)

Этот каталог и файлы `docker-compose.yml` / `.env.example` в корне репозитория
позволяют развернуть **серверную часть** системы — REST API (`CarCare360.Api`)
и базу данных Microsoft SQL Server — **одной командой**, без ручной установки
SQL Server и ручного применения SQL-скриптов.

> Доработка не изменяет структуру базы данных: контейнер инициализации применяет
> существующие скрипты `SQL/01..03` без модификации схемы.

## Что входит в стенд

| Сервис | Назначение |
|--------|-----------|
| `db` | Microsoft SQL Server (том `mssql-data` хранит данные между перезапусками). |
| `db-init` | Одноразовый контейнер: создаёт БД `CarCare360` и применяет `SQL/01→02→03`. |
| `api` | Серверный REST API (внутри слушает порт 8080, наружу — `API_PORT`). |

## Требования

- Docker Engine 24+ / Docker Desktop с плагином Docker Compose v2;
- свободно ≈ 6 ГБ на диске (образы + том данных СУБД);
- доступ в интернет для первичной загрузки базовых образов.

## Шаги развёртывания

1. Скопируйте шаблон переменных окружения и задайте значения:

   ```bash
   cp .env.example .env
   ```

   В `.env` обязательно смените:
   - `MSSQL_SA_PASSWORD` — пароль `sa` (мин. 8 символов, минимум 3 категории из
     4: заглавные, строчные, цифры, спецсимволы);
   - `JWT_SIGNING_KEY` — длинный случайный секрет подписи JWT.

   Пример заполненного `.env` (значения иллюстративные, замените своими):

   ```dotenv
   MSSQL_SA_PASSWORD=Str0ng_Pass!2026
   MSSQL_PID=Express
   DB_PORT=1433
   API_PORT=5009
   ASPNETCORE_ENVIRONMENT=Development
   JWT_SIGNING_KEY=7f3c9e1a2b8d4f6a0c5e9b7d1f3a6c8e0b2d4f6a8c1e3b5d7f9a1c3e5b7d9f10
   ```

2. Поднимите стенд из **корня репозитория**:

   ```bash
   docker compose up -d --build
   ```

3. Дождитесь готовности. Полный цикл «с нуля» (загрузка образов, сборка API,
   инициализация БД) — обычно несколько минут.

## Проверка работоспособности

- Статус контейнеров:

  ```bash
  docker compose ps
  ```

  `db` — `healthy`; `carcare360-db-init` — `Exited (0)` (одноразовый, это норма);
  `api` — `running`.

- Логи инициализации базы данных (должна быть строка об успехе):

  ```bash
  docker compose logs db-init
  ```

  Ожидаемо: `[db-init] Инициализация базы данных завершена успешно.`

- API доступен по сети. При `ASPNETCORE_ENVIRONMENT=Development` открыта
  страница Swagger:

  ```
  http://localhost:5009/swagger
  ```

  Либо публичный эндпоинт каталога услуг (не требует токена), например:

  ```bash
  curl http://localhost:5009/api/services
  ```

## Как подключаются десктоп и мобильное приложение

- Порт API на хосте — `API_PORT` (по умолчанию **5009**), как и при локальном
  запуске. Дополнительно настраивать клиентов не нужно:
  - мобильное приложение на эмуляторе — `http://10.0.2.2:5009`;
  - десктоп — строка подключения к БД в `App.config`. Чтобы десктоп работал с
    контейнерной БД, укажите в нём `Server=localhost,<DB_PORT>` (по умолчанию
    `localhost,1433`), `User Id=sa`, тот же пароль и `TrustServerCertificate=True`.

## Управление стендом

```bash
docker compose logs -f api      # логи API
docker compose restart api      # перезапуск API без пересборки
docker compose down             # остановить и удалить контейнеры (том сохраняется)
docker compose down -v          # то же + удалить том с данными БД (ПОЛНЫЙ сброс)
```

## Возможные проблемы

- **`MSSQL_SA_PASSWORD` не задан / слишком простой** — контейнер `db` не
  стартует и завершает работу. Задайте надёжный пароль в `.env`.
- **Порт `API_PORT`/`DB_PORT` занят** — измените значение в `.env` и повторите
  `docker compose up -d`.
- **Повторный запуск** не пересоздаёт схему: `db-init` видит существующую таблицу
  `Roles` и пропускает скрипты `01`/`02` (идемпотентность).
- **Сменить окружение API**: установите `ASPNETCORE_ENVIRONMENT=Production` в
  `.env` (Swagger при этом будет отключён) и выполните `docker compose up -d`.
