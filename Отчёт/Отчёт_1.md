Реализованы следующие виды работ:

постановка задачи и уточнение требований к программному продукту на основе технического задания; проектирование реляционной модели данных (роли, пользователи, марки и модели автомобилей, автомобили, специализации, механики, рабочие посты, услуги, запчасти, заказ-наряды и их детализация по услугам и запчастям, журнал аудита, refresh-токены), связей и ограничений целостности; разработка SQL-скрипта создания базы данных, триггеров аудита и хранимых процедур (создание заказа, смена статуса, добавление услуги, атомарное списание запчасти с контролем остатка); реализация серверной части на языке C# с использованием Entity Framework Core (модели предметной области, контекст CarCareDbContext, работа с данными через ORM и хранимые процедуры); реализация архитектурного шаблона MVVM в десктопном приложении (модели представления, команды, привязка данных) и реализации восьми функциональных модулей (клиенты, автомобили, заказы, склад, механики, справочники, аудит, отчёты); реализация системы аутентификации с хешированием паролей по алгоритму BCrypt (хранение в BINARY(64)) и ролевой модели доступа; разработка серверного REST API с JWT-аутентификацией (access- и refresh-токены с ротацией), разграничением доступа по ролям, ограничением частоты запросов и документацией Swagger; разработка мобильного приложения для клиентов на платформе Flutter с управлением состоянием Riverpod, HTTP-клиентом Dio и JWT-интерцептором, авто-входом, экранами автомобилей, записи на сервис, истории заказов, профиля с локальным аватаром и push-уведомлениями (Firebase Cloud Messaging); реализация отчётности с дашбордом и графиками (LiveCharts2); отладка запросов и обработка ошибок; сквозное тестирование сценариев пользователя и сотрудника; сборка и публикация приложений; оформление технической и пользовательской документации.

При прохождении учебной практики я понял, что разработка программного обеспечения для компьютерных систем — это не только написание кода, но и последовательное прохождение этапов от анализа требований до проверки результата. Особенно это заметно в проекте из нескольких компонентов: десктопное приложение, серверный API и мобильный клиент должны согласованно работать с одной базой данных и одинаково понимать форматы данных и бизнес-правила (например, статусы заказа и совместимый формат хранения хэша пароля). Ошибки на ранних этапах проектирования схемы данных затем дорого исправляются, поэтому имеет смысл заранее фиксировать состав сущностей, связи и сценарии использования до масштабной реализации.

Практика помогла мне закрепить навыки чтения технической документации, пошаговой сборки приложения из отдельных частей (моделей, сервисов, представлений, контроллеров) и проверки работоспособности после каждого логически завершённого шага. Я освоил архитектурный шаблон MVVM в WPF, работу с Entity Framework Core и SQL Server, разработку REST API с JWT-аутентификацией, а также разработку мобильного приложения на Flutter с управлением состоянием Riverpod и взаимодействием с сервером по HTTP. Научился применять механизмы безопасности (хеширование паролей, ролевое разграничение доступа, ограничение частоты запросов, хранение токенов в защищённом хранилище) и формулировать задачу так, чтобы её можно было разбить на подзадачи и выполнить в отведённые сроки.

При выполнении задач практики самым трудным оказалось согласовать работу разных компонентов системы: чтобы данные, введённые сотрудником в десктопном приложении, корректно сохранялись в базе, отдавались через API и отображались в мобильном приложении клиента, нужно было внимательно проверять типы полей, ограничения базы данных, форматы JSON и обработку исключительных ситуаций.
