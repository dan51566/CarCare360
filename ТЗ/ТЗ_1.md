# 1. ВВЕДЕНИЕ

## 1.1. Наименование программы

Наименование — Комплексная информационная система автосервиса «CarCare 360».

Система включает три компонента:
- `CarCare360.Desktop` — десктопное приложение для сотрудников (WPF, .NET 10);
- `CarCare360.Api` — серверный REST API (ASP.NET Core 10);
- `carcare_mobile` — мобильное приложение для клиентов (Flutter, Android).

## 1.2. Краткая характеристика области применения программы

Программа предназначена для автоматизации деятельности станции технического обслуживания (автосервиса): учёта клиентов и их автомобилей, оформления и сопровождения заказ-нарядов на ремонт и обслуживание, ведения склада запчастей, учёта механиков и распределения работ, ведения справочной информации, журналирования изменений и формирования отчётности.

Сотрудник автосервиса (администратор, механик) работает с десктопным приложением: ведёт клиентскую базу, автомобили, создаёт и сопровождает заказ-наряды (с назначением услуг, механиков, рабочих постов и списанием запчастей), управляет складом, справочниками, просматривает журнал аудита и отчёты. Клиент автосервиса работает с мобильным приложением: регистрируется, добавляет свои автомобили, самостоятельно записывается на сервис, отслеживает статус заказов и получает push-уведомления. Связь между мобильным приложением и общей базой данных обеспечивает серверный REST API.

Объект автоматизации: процессы регистрации клиентов и автомобилей, оформления и обработки заказ-нарядов, контроля остатков запчастей на складе, распределения работ между механиками, ведения журнала изменений критичных данных и информирования клиентов о ходе обслуживания.

# 3. НАЗНАЧЕНИЕ РАЗРАБОТКИ

## 3.1. Функциональное назначение программы

Система должна обеспечивать следующий функционал, распределённый по трём компонентам.

### 3.1.1. Десктопное приложение для сотрудников (CarCare360.Desktop)

**Авторизация и доступ:**
1. Вход в систему по паре логин / пароль (пароль хранится в виде BCrypt-хэша); функция «Запомнить меня»; выход из системы.
2. Регистрация новых учётных записей.
3. Ролевая модель доступа: состав видимых разделов зависит от роли пользователя (см. п. 3.2).

**Модуль «Клиенты»:**
4. Список клиентов с поиском по имени, телефону, e-mail; создание и редактирование профиля клиента; регистрация новых клиентов; просмотр автомобилей и истории заказов клиента; мягкое удаление (флаг `IsDeleted`); хранение аватара пользователя.

**Модуль «Автомобили»:**
5. Привязка автомобиля к клиенту; ввод полей: марка/модель (из справочника), год выпуска, VIN, государственный номер, цвет, пробег; CRUD-диалоги.

**Модуль «Заказы» (заказ-наряды):**
6. Создание заказ-наряда с выбором автомобиля, даты и времени.
7. Статусная машина из 7 состояний: «Новый» → «Назначен» → «В работе» → «Ожидает запчасти» → «Готов» → «Выдан» → «Отменён».
8. Детализация заказа: список услуг (с назначением механика и рабочего поста) и список запчастей; добавление услуг и запчастей; контроль остатка запчастей на складе при списании (хранимая процедура с транзакцией).
9. Блокировка редактирования заказа, находящегося в финальном статусе («Готов», «Выдан», «Отменён»): кнопки смены статуса и сохранения шапки заблокированы, отображается информационный баннер.

**Модуль «Склад»:**
10. Каталог запчастей с артикулом, ценой и количеством; визуальный индикатор низкого остатка; атомарное списание запчасти при добавлении в заказ; CRUD.

**Модуль «Механики»:**
11. Список механиков с фото, специализацией и уровнем квалификации; назначение механика на услугу внутри заказа; корректный подсчёт числа заказ-нарядов, в которых задействован механик (по идентификатору `MechanicID`); создание и редактирование.

**Модуль «Справочники»:**
12. Управление марками и моделями автомобилей, видами услуг (наименование, нормо-час, базовая цена), специализациями механиков, рабочими постами (боксами).

**Модуль «Аудит»:**
13. Просмотр и фильтрация журнала аудита: имя таблицы, тип операции (I/U/D), первичный ключ записи, пользователь, дата изменения, старое и новое значение в формате JSON.

**Модуль «Отчёты»:**
14. Финансовый дашборд: ключевые показатели (количество заказов, выручка, новые клиенты, активные механики, складские показатели) за выбранный период; графики — столбчатый график выручки по дням, круговая диаграмма структуры доходов (услуги/запчасти), линейный график среднего чека (библиотека LiveCharts2); сводка нагрузки механиков. Экспорт в Excel — заглушка (запланировано в следующей итерации).

### 3.1.2. Серверный REST API (CarCare360.Api)

15. Аутентификация: вход, регистрация клиента, обновление токена (ротация refresh-токена), выход; выдача пары access/refresh JWT-токенов.
16. Предоставление эндпоинтов для мобильного клиента: автомобили (CRUD), заказы (создание, просмотр, отмена), справочник услуг (публичный), справочник марок и моделей (публичный), профиль клиента, отчёты, журнал аудита, механики, склад.
17. Разграничение доступа по ролям: клиент видит только свои данные; механик — назначенные ему заказы; администратор — все данные.
18. Ограничение частоты запросов к эндпоинтам аутентификации (защита от подбора).
19. Единый формат обработки ошибок (middleware) и документация Swagger (только в режиме разработки).

### 3.1.3. Мобильное приложение для клиентов (carcare_mobile)

20. Авторизация: экран входа и регистрации; автоматический вход при запуске (токен хранится в защищённом хранилище, при истечении access-токена автоматически вызывается обновление); выход.
21. Автомобили: список автомобилей клиента; добавление с выбором марки/модели из справочника, вводом госномера, VIN, года, цвета, пробега; редактирование и удаление.
22. Запись на сервис: выбор автомобиля, даты и времени, мультивыбор услуг из каталога с ориентировочной стоимостью, комментарий; создание заказа со статусом «Новый».
23. История заказов: список заказов (новые сверху), карточка с цветовой индикацией статуса, детализация по услугам и запчастям; отмена записи при статусах «Новый» и «Назначен» (мягкое удаление через DELETE).
24. Профиль: просмотр и редактирование (ФИО, e-mail, телефон); аватар, выбираемый из галереи или камеры и хранящийся локально на устройстве; смена пароля — заглушка (не поддерживается API); выход из аккаунта.
25. Уведомления: интеграция с Firebase Cloud Messaging (push-уведомления при наличии конфигурации); локальные уведомления; экран истории уведомлений.

### 3.1.4. Общесистемные требования к реализации (проект CARCARE360)

- Десктопное приложение — архитектурный шаблон MVVM (Model-View-ViewModel) на платформе WPF / .NET 10.
- Серверный API — тонкий транспортный слой над общей базой данных; бизнес-логика в слое сервисов; критичные операции (создание заказа, смена статуса, списание запчастей) — через хранимые процедуры SQL Server.
- Мобильное приложение — управление состоянием Riverpod, HTTP-клиент Dio с JWT-интерцептором (автоматическое обновление токена при ответе 401), навигация go_router, сериализация json_serializable.
- Реляционная модель данных через Entity Framework Core; СУБД — Microsoft SQL Server (экземпляр `.\SQLEXPRESS`, база `CarCare360`).
- Десктоп и API используют **одну и ту же** базу данных.
- Хэширование паролей — BCrypt.Net-Next (work factor 12), хранение хэша в поле `PasswordHash BINARY(64)` (совместимость десктопа и API).
- Журнал аудита — SQL-триггеры на таблицах `Users`, `Cars`, `Orders` (операции INSERT/UPDATE/DELETE с JSON-снимками старых и новых значений).
- Ролевая модель доступа: роли «Администратор», «Механик», «Клиент». Мобильное приложение предназначено только для роли «Клиент».
- Мягкое удаление (`IsDeleted`) для пользователей и заказов; для автомобилей — жёсткое удаление с защитой от удаления при наличии связанных заказов.
- Аутентификация API — JWT Bearer (access + refresh токены, ротация); аутентификация десктопа — синглтон текущего пользователя.

| Роль | Компонент | Описание |

| Администратор | Десктоп | Полный доступ ко всем 8 модулям: клиенты, автомобили, заказы, склад, механики, справочники, аудит, отчёты. |
| Механик | Десктоп | Клиенты, автомобили, заказы, механики; скрыты: склад, справочники, аудит, отчёты. |

Мобильное приложение — на смартфонах и планшетах клиентов под управлением Android. Серверный API размещается на сервере (или учебном стенде) и доступен мобильным клиентам по сети.

# 4. ТРЕБОВАНИЯ К ПРОГРАММЕ

## 4.1. Требования к функциональным характеристикам

### 4.1.1. Требования к составу выполняемых функций

Структура хранимых данных (единая база `CarCare360`):

| Таблица | Ключевые поля и ограничения |
|---------|------------------------------|
| Roles | Name до 50 символов, уникальный; справочник ролей (Администратор, Механик, Клиент). |
| Users | Login до 50, уникальный; PasswordHash BINARY(64); FullName до 100; Email до 100; Phone до 20; RoleId (внешний ключ на Roles); IsActive; IsDeleted; CreatedAt. |
| CarBrands | Name до 50, уникальный; Country до 50. |
| CarModels | BrandId (внешний ключ на CarBrands); Name до 100; YearFrom, YearTo; уникальность (BrandId, Name). |
| Cars | ClientId (внешний ключ на Users); ModelId (внешний ключ на CarModels); Year; VIN до 17; LicensePlate до 9, обязательно; Color; Mileage; индексы по VIN и LicensePlate. |
| Specializations | Name до 80, уникальный. |
| Mechanics | UserId (внешний ключ на Users, уникальный); SpecializationId; HireDate; QualificationLevel. |
| ServiceBays | Name до 20, уникальный; IsActive. |
| Services | Name до 100; Description до 200; NormHour decimal(4,2), обязательно; BasePrice decimal(10,2). |
| Parts | Name до 100; PartNumber до 50, уникальный; QuantityInStock; Price decimal(10,2). |
| Orders | CarId, ClientId (внешние ключи); CreatedAt по умолчанию GETDATE(); ScheduledDate; ScheduledTime; Status — ограничение CHECK на 7 значений, по умолчанию «Новый»; Mileage; Notes до 500; IsDeleted; индекс по Status. |
| OrderServices | OrderId, ServiceId (внешние ключи); MechanicId (внешний ключ на Mechanics); BayId; StartTime, EndTime; Status — CHECK (Назначена / В работе / Выполнена / Отменена). |
| OrderParts | OrderId, PartId (внешние ключи); Quantity, CHECK (>0); PricePerUnit decimal(10,2). |
| AuditLog | TableName до 50; Operation 1 символ (I/U/D); PrimaryKeyValue; ChangedBy; ChangedAt по умолчанию GETDATE(); OldValues, NewValues — текст JSON; индексы по TableName и ChangedAt. |
| RefreshTokens | Token, уникальный; UserId (внешний ключ); срок действия и отзыв (используется только серверным API). |

Хранимые процедуры: `RegisterClient`, `CreateOrder`, `AddServiceToOrder`, `UpdateOrderStatus`, `AddPartToOrder` (атомарное списание запчасти в транзакции).

Триггеры аудита: `trg_Audit_Users`, `trg_Audit_Cars`, `trg_Audit_Orders` (AFTER INSERT/UPDATE/DELETE).

Операции CRUD над справочниками, складом, смена статусов заказов, просмотр аудита и отчётов доступны только пользователю с ролью «Администратор» (часть функций — также механику согласно п. 3.2). Создание заказов и автомобилей доступно клиенту через мобильное приложение в пределах своих данных.

### 4.1.2. Требования к организации входных данных

Входные данные поступают:

1. Через интерфейс десктопного приложения: ввод с клавиатуры (логин, пароль, данные клиентов, автомобилей, заказов, запчастей, справочников), выбор значений из выпадающих списков и таблиц, выбор файлов изображений (аватары).
2. Через интерфейс мобильного приложения: ввод данных регистрации и профиля, параметров автомобиля, параметров записи на сервис, выбор услуг, выбор изображения аватара из галереи/камеры.
3. Через HTTP-запросы мобильного клиента к API: тело запросов в формате JSON (вход, регистрация, создание автомобиля, создание заказа, смена статуса, редактирование профиля); параметры строки запроса (фильтры отчётов и аудита, пагинация); заголовок `Authorization: Bearer <token>`.

На стороне сервера и приложений должна выполняться валидация: обязательность заполнения полей, длины строк, уникальность логина при регистрации, достаточность остатка запчасти при списании, допустимость статуса заказа, корректность форматов даты и времени, проверка прав доступа к запрашиваемым данным.

### 4.1.3. Требования к организации выходных данных

1. Для сотрудника (десктоп): окна и таблицы с данными (клиенты, автомобили, заказы, склад, механики, аудит); диалоги создания/редактирования; отчёты с числовыми показателями и графиками; всплывающие уведомления (toast) об успехе/ошибке; анимированные переходы между разделами.
2. Для клиента (мобильное приложение): экраны с адаптивной раскладкой; список и карточки автомобилей и заказов с цветовой индикацией статуса; формы с сообщениями об успехе и об ошибках (SnackBar); локальные и push-уведомления.
3. Для мобильного клиента (API): ответы в формате JSON (camelCase с сохранением акронимов: `userID`, `carID`, `orderID`); даты в формате ISO 8601; единый формат ошибок `{ "error": "...", "detail": "..." }` с соответствующими HTTP-кодами (400/401/403/404/409/429).
4. Графики отчётов формируются на стороне десктопного приложения средствами библиотеки LiveCharts2.

### 4.1.4. Требования к временным характеристикам

- Отклик навигации между разделами десктопного приложения — менее 100 мс; длительность анимации перехода — около 300 мс.
- Типовой запрос к базе данных (список записей) — менее 200 мс; скелетная загрузка отображается при ожидании свыше 150 мс.
- Открытие списка заказов при объёме до нескольких тысяч записей — до 3 секунд.
- Операция оформления заказа и списания запчастей выполняется в транзакции; суммарное время — не более 5 секунд при нормальной нагрузке учебного стенда.
- Время жизни access-токена API — 15 минут, refresh-токена — 7 дней; обновление токена выполняется прозрачно для пользователя мобильного приложения.

Обеспечивается комплексом мероприятий:

### 4.2.2. Время восстановления после отказа

- После потери данных — время восстановления из резервной копии зависит от объёма копии и административных действий по развёртыванию.

### 4.2.3. Отказы из-за некорректных действий оператора

Отказ программного обеспечения возможен вследствие некорректных действий оператора при взаимодействии с операционной системой или попытках прямого изменения структуры базы данных. Во избежание отказов реализована ролевая модель доступа (RBAC): клиенты не имеют доступа к данным других клиентов и к административным функциям; механикам недоступны склад, справочники, аудит и отчёты; в API доступ к чужим данным возвращает код 403. Запросы аутентификации защищены ограничением частоты.

- Аутентификация API — JWT Bearer: access-токен (15 минут), refresh-токен (7 дней) с ротацией при обновлении, алгоритм HMAC-SHA256, допуск рассинхронизации часов 30 секунд.
- Эндпоинты аутентификации защищены ограничением частоты запросов (алгоритм SlidingWindow, 3 запроса в минуту с одного IP; при превышении — код 429).
- Токены на мобильном устройстве хранятся в защищённом хранилище (flutter_secure_storage).
- Все операции с базой данных выполняются через EF Core и параметризованные хранимые процедуры — SQL-инъекции исключены.
- Ролевая модель доступа: десктоп фильтрует доступные разделы по роли; API возвращает 403 при попытке доступа к чужим данным.
- Мягкое удаление сохраняет данные для аудита и восстановления; журнал аудита фиксирует все изменения критичных таблиц.
- В режиме разработки для мобильного клиента разрешён обычный HTTP (`usesCleartextTraffic`); в защищённой эксплуатации передача данных осуществляется по HTTPS, ключ подписи JWT и строка подключения хранятся вне репозитория.

# 8. ПОРЯДОК КОНТРОЛЯ И ПРИЁМКИ

## 8.1. Виды испытаний

Контроль и приёмка осуществляются руководителем практики / преподавателем по результатам представления следующих материалов и демонстрации.

**Комплект материалов:**
- исходный код (репозиторий или архив) трёх компонентов;
- настоящее Техническое задание;
- отчёт и дневник;
- краткая инструкция по запуску (настройка строки подключения, создание базы и начальное заполнение, запуск API, установка и запуск мобильного приложения).

**Контрольная демонстрация функционала:**

1. Запуск десктопного приложения; вход под администратором (`admin` / `admin123`); проверка отображения всех 8 модулей.
2. Вход под учётной записью механика; проверка скрытия разделов «Склад», «Справочники», «Аудит», «Отчёты».
3. Создание клиента и автомобиля в десктопном приложении; проверка появления записей в списках и в журнале аудита.
4. Создание заказ-наряда; добавление услуги с назначением механика и поста; добавление запчасти — проверка уменьшения остатка на складе; попытка списать больше остатка — проверка отклонения операции.
5. Проверка блокировки редактирования заказа в финальном статусе («Готов» / «Выдан» / «Отменён»).
6. Просмотр отчётов: проверка отображения KPI, графиков (выручка, структура доходов, средний чек) и сводки нагрузки механиков.
7. Запуск серверного API; проверка доступности (Swagger в режиме разработки, публичный эндпоинт услуг).
8. Запуск мобильного приложения на эмуляторе/устройстве; вход под учётной записью клиента; проверка отображения его автомобилей и заказов.
9. Сквозной сценарий: в десктопном приложении (под администратором) создаётся клиент и автомобиль → в мобильном приложении клиент входит и видит автомобиль → клиент создаёт заказ → в десктопном приложении изменяется статус заказа → клиент видит обновлённый статус в мобильном приложении.
10. Отмена записи клиентом в мобильном приложении (при статусе «Новый» / «Назначен») — проверка мягкого удаления заказа.
11. Неуспешная попытка входа с неверным паролем; проверка ограничения частоты запросов к эндпоинту входа API; попытка доступа к чужим данным через API — возврат кода 403.

**Оценка качества:** выполнение работ считается удовлетворительным, если реализованный функционал и документы соответствуют требованиям настоящего Технического задания; при наличии отклонений составляется перечень доработок.
